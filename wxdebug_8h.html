Detailed Description

Debugging Library Header.

This library provides straightforward debugging functionality, split into two main sections.

The first section is assertion handling, which includes three types of assertions:

  • The most commonly used is the ASSERT(condition) macro, which pops up a message box including the file and line number if the condition evaluates to FALSE.
  • The EXECUTE_ASSERT macro functions similarly to ASSERT, but the condition will still be executed in NON-debug builds.
  • The KASSERT macro is more suitable for pure (perhaps kernel) filters, as the condition is printed onto the debugger rather than in a message box.

The second part of the debug module facilities is general-purpose logging, accessed by calling DbgLog(). This function takes a type and level field that define the type of informational string being presented and its relative importance. The type field can be a combination of one or more of the following:

  • LOG_TIMING
  • LOG_TRACE
  • LOG_MEMORY
  • LOG_LOCKING
  • LOG_ERROR

The level is a DWORD value where zero defines the highest importance. Use of zero as the debug logging level is encouraged ONLY for major errors or events, as they will ALWAYS be displayed on the debugger. Other debug output has its level matched against the current debug output level stored in the registry for this module, and if it is less than the current setting, it will be displayed.

Each module or executable has its own debug output level for each of the five types. These levels are read when the DbgInitialise function is called. For DLLs linking to STRMBASE.LIB, this is done automatically when the DLL is loaded. Executables must call DbgInitialise explicitly with the module instance handle given to them through the WINMAIN entry point. An executable must also call DbgTerminate when it has finished to clean up the resources the debug library uses; this is done automatically for DLLs.

These are the five different categories of logging information.
